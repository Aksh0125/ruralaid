1. Executive Summary

RuralHealthConnect is an asynchronous teleconsultation platform designed to bridge the healthcare access gap in rural and semi-urban India. The platform connects patients in remote areas with qualified doctors through a mobile-first web application, eliminating the need for real-time connectivity or physical travel.

Patients describe their illness in text (in any language — Hindi, Hinglish, or English), the system automatically matches their request to nearby specialist doctors using geospatial querying, a doctor reviews and submits a structured treatment plan, and the patient pays a nominal fee to unlock and view the medical advice.

Key differentiators:

  • Asynchronous model — works on low-bandwidth connections
  • Multi-language input with automatic English translation for matching
  • Geographic-aware doctor matching using PostGIS
  • End-to-end medical data encryption (AES-256-GCM)
  • Video prescription add-on — doctors can record a video explaining their prescription
  • Rural zone classification for priority routing

2. Problem Statement

Rural India faces a severe shortage of accessible healthcare:

  • Over 600 million people in rural areas have limited access to specialist doctors
  • Nearest specialist may be 50–200 km away, requiring travel that costs more than consultation
  • Poor internet connectivity makes real-time video consultations unreliable
  • Low health literacy makes it difficult for patients to articulate symptoms in English
  • Doctors in small towns have under-utilized capacity but no discovery mechanism

RuralHealthConnect solves this by:

  • Enabling asynchronous text-based consultation — no real-time connection needed
  • Accepting illness descriptions in Hindi/Hinglish and auto-translating for matching
  • Using GPS to find the geographically closest matching doctor
  • Making consultations affordable (pay-per-unlock model)

3. System Overview & Architecture

The system follows a three-tier architecture: a React web frontend, a Node.js/Express REST API backend, and a PostgreSQL + PostGIS database with Redis for ephemeral state.

4. Component Design

🔐 Auth Service

Handles patient/doctor registration, OTP lifecycle (6-digit, 10-min TTL, 3 attempts), JWT issuance (24h expiry). OTP stored in Redis. SMS via Twilio/MSG91.

👤 Patient Service

Patient profile management, consultation history (paginated, reverse-chronological), account deletion with PII anonymization (30-day hard delete).

👨‍⚕️ Doctor Service

Doctor profile CRUD, specialization management (1–5 tags), availability status updates (30s SLA), geographic service area with PostGIS storage, consultation queue management.

📋 Consultation Service

Consultation request creation, status machine (PENDING → ACCEPTED → TREATMENT_READY → UNLOCKED), role-aware serialization (strips PII from doctor view), atomic acceptance.

🗺 Matcher Service

BullMQ worker. Extracts specialization keywords from illness description, queries doctors via PostGIS ST_DWithin, ranks by distance, forwards to top-10. Expands radius 50→100→150→200km. Falls back to any available doctor.

💊 Treatment Plan Service

Structured treatment plan (diagnosis + steps + medications), AES-256-GCM encryption at rest, immutable after submission, optional video prescription upload to S3.

💳 Payment Service

Razorpay order creation, HMAC-SHA256 webhook verification, idempotent payment confirmation, consultation unlock on success, payment receipt generation, 3-attempt retry limit.

🔔 Notification Service

Firebase Cloud Messaging (FCM) push notifications. Notifications: forwarded request (doctor), accepted (patient), treatment ready (patient), payment confirmed (patient). Offline queuing via FCM.

5. Data Flow & Consultation Lifecycle

Full Consultation Flow

1
Patient Registration: Patient registers with name, phone (E.164), DOB, gender, GPS location. OTP sent to phone → verified → JWT issued.
2
Consultation Submission: Patient types illness description (20–2000 chars) in any language. System auto-translates to English, captures GPS coordinates, creates Consultation_Request with status = PENDING.
3
Doctor Matching: Matcher BullMQ worker extracts specialization keywords, queries PostGIS for doctors within 50km, expands to 500km if none found, falls back to any available doctor. Creates Consultation_Forward records for top-10 ranked doctors.
4
Doctor Notification: FCM push notification sent to matched doctors with first 200 chars of illness description. Doctor sees the request in their dashboard (only district shown, no patient PII).
5
Doctor Accepts: Doctor accepts consultation. Status → ACCEPTED atomically. Patient notified via FCM. No other doctor can accept same request.
6
Treatment Plan Submission: Doctor fills structured form: diagnosis (50–2000 chars), treatment steps (1–20), medications (optional), optional video prescription. All fields encrypted with AES-256-GCM. Status → TREATMENT_READY. Record marked immutable.
7
Patient Pays: Patient sees paywall. Initiates Razorpay payment (UPI/mobile money). On confirmation, status → UNLOCKED. Treatment plan decrypted and shown. Signed S3 URL for video if present.
8
History Access: Patient can always access unlocked treatment plans from consultation history. No re-payment required.

PostGIS Geo-Query Pattern

Doctor matching uses ST_DWithin with a GIST index on service_area_center for sub-500ms queries on 100k+ doctor profiles:

SELECT d.id, ST_Distance(d.service_area_center, patient_point) / 1000 AS distance_km
FROM doctors d
JOIN doctor_specializations ds ON ds.doctor_id = d.id
WHERE d.availability_status = 'ACTIVE'
  AND d.account_status = 'APPROVED'
  AND ds.specialization_tag = ANY($specialization_tags)
  AND ST_DWithin(d.service_area_center, patient_point, $radius_meters)
ORDER BY distance_km ASC LIMIT 10;

8. Security Design

Security LayerImplementation
AuthenticationPhone OTP (6-digit, 10-min TTL, 3 attempts max) + JWT Bearer tokens (24h expiry, HS256)
AuthorizationRole-Based Access Control (PATIENT / DOCTOR). Middleware enforces role on every protected endpoint. Patients cannot access other patients' data.
Data Encryption at RestAES-256-GCM for all Treatment Plan fields (diagnosis, steps, medications). Key stored in environment / KMS. IV + auth tag packed with ciphertext in base64.
Data Encryption in TransitTLS 1.2+ enforced on all client-server communication. HTTPS-only in production.
Patient PII ProtectionDoctors never see patient name, phone, DOB, or exact coordinates. Only administrative district shown. Role-aware serializer strips PII fields before sending response.
Payment SecurityRazorpay HMAC-SHA256 webhook signature verification. Idempotent payment processing (checked by razorpay_payment_id uniqueness).
Video Access ControlPrescription videos stored as private S3 objects. Access via time-limited signed URLs (1-hour expiry) generated server-side only for authenticated, payment-confirmed patients.
PII LifecycleOn account deletion: immediate anonymization of all linked records. Hard deletion scheduled within 30 days via BullMQ with exponential backoff retry.
Security HeadersHelmet.js: Strict-Transport-Security, X-Content-Type-Options, X-Frame-Options. CORS restricted to frontend origin.
Input ValidationAll fields validated at API layer before DB insert. Field-level errors returned to client. SQL injection prevented by parameterized queries throughout.

9. Add-On Features

🌐 Multi-Language Input

Patients can describe symptoms in Hindi, Hinglish, or English. The backend auto-detects language and translates to English using Google Translate API before keyword extraction and doctor matching. Original text preserved in logs.

🎥 Video Prescription

Doctor can record or upload a short video (MP4/MOV, max 2 min, 100MB) before submitting the treatment plan. Video stored privately in AWS S3. Accessible to patient only after payment confirmation via time-limited signed URL.

🗺 Rural Zone Tagging

Background BullMQ job tags each doctor with the rural zone they serve using PostGIS ST_Contains check against stored boundary polygons. Enables priority routing to doctors serving classified rural areas.

⏰ Forwarding Expiry

If a doctor doesn't respond within 24 hours, the forward is marked expired and the request is automatically forwarded to the next ranked candidate doctor. Background job runs every 5 minutes.
